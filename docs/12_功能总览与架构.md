# AetherNarrator 功能总览与架构（沉淀文档）

> 本文档沉淀 C 项目（零后端浏览器端 AI 文字冒险 / 单人跑团引擎）**截至当前已实现的功能**，作为后续开发与交接的索引。
> 更细的单点设计见 `docs/01`~`docs/11`。

---

## 0. 项目定位与运行模型

- **零后端**：纯前端，浏览器运行。API Key 存于 `localStorage`，由玩家自填。
- **多 `<script>` 共享词法全局**：`app-core` / `app-ai` / `app-game` / `map-data` / `combat-*` / `action-menu` / `renderer/tile-map` / `app-ui` 等文件在页面中按序加载，函数与 `var` 声明挂到全局（如 `gameState`、`currentWorld`）。
- **确定性测试体系**：`test/` 下用 Node `vm` 沙箱加载**真实源码**做不耗 API、不依赖 DOM 的单元测试（详见 §12）。

## 1. 世界与世界观生成

- 创建入口 `callWorldGenerationLLM` 一次性产出：`schema`（进度/技能/属性中文映射/时间模式）、`initial_state`（主角初始能力、关系、物品、目标、npc_states…）、`lore_kb`（知识库）、`opening_narrative`、`system_prompt`。
- 两种类型：原创世界 / 基于已有 IP（如哈利波特）。支持规则集（dnd 等）、规则自由度（1-5）、世界自由度（1-5）、文风参考。
- 主角能力**不可降级**：初始状态必须如实反映设定，AI 不得擅自降为"初学者"。

### 1.1 一键导入 IP 设定（与"上传源文件"互补，不冲突）

- **定位差异**：创建世界时「上传 TXT/DOCX/EPUB 源文件」是让 AI **生成整套世界**的原料（原文仅存 `source_content` 备查，运行时不注入 RAG）；「一键导入 IP 设定」则是把原著**抽取为结构化知识片段**并挂载进 `lore_kb`，运行时 RAG 每轮命中，约束叙事一致性。
- **入口与流程**：世界详情弹窗「导入 IP 设定」→ 粘贴文本 → `extractLoreFromText`（模拟模式离线可用，不耗 API）→ 预览片段 → 确认 `mergeLoreSnippets`（按 title 去重）→ 写入 `lore_kb` 并触发 `computeEmbeddingsForWorld` 重算向量。
- **预设世界只读保护** `ensureEditableWorld`：预设世界（`preset === true`，`system_prompt` 含写死 canon）导入前自动派生为「可编辑副本」，避免污染原 canon；自定义世界直接导入。
- **版权开关 UI（二期已落地）**：导入弹窗提供「① 版权状态」选项——`公版/原创（可分享）` 与 `受版权（仅本地自用）`。受版权时 `updateLoreCopyrightHint` 实时将提示文案置红并警告；`confirmImportLore` 写入 `lore_copyright`（'safe'/'copyrighted'）与 `local_only`（受版权=true）。世界详情页据此显示徽章（公版/原创→绿色"允许分享"；受版权→红色"仅限本地自用，请勿分享或导出"），为未来"导出/分享世界"功能预留 `local_only` 边界；预设库（§10.6）维持公版/原创，不收录受版权 IP。
- **精修/覆盖模式（二期已落地）**：导入弹窗提供「② 导入模式」——`追加+去重`（默认，调 `mergeLoreSnippets` 保留原片段并按 title 去重）/ `覆盖同类`（调 `replaceLoreByCategory`，删除 base 中与 incoming 同 category 的旧片段再追加去重）/ `完全覆盖`（调 `overwriteLoreSnippets`，清空知识库仅用本次导入）。三者均经 `normalizeLoreSnippets` 规整字段与补 id。
- **长文分块抽取（二期已落地）**：原文 ≤ `LORE_CHUNK_SIZE`(9000) 时单次抽取；超长则 `splitLoreChunks` 按**段落边界 + 块间重叠(1500)**切分（重叠量被钳制为 ≤ 单块 1/3，防步长退化），逐块 `extractOneChunk` 真实抽取后 `mergeLoreSnippets` 去重合并；`LORE_MAX_CHUNKS`(12) 限制总调用数。分块 prompt 标注「第 x/y 段」并约束只抽本段设定，避免跨段臆造。UI 长文抽取时实时刷新「正在抽取设定…（第 x/y 块）」。
- **测试**：`test/import_lore_check.js`（48 PASS，含分块逻辑、长文 mock 抽取、版权标记、三种导入模式分支）；`test/import_lore_live.js`（真实 API 联调 32 PASS，验证单块抽取、分块合并≥单次、预设派生、受版权 local_only、覆盖同类全链路）。

### 1.2 NPC 地图化（叙事位置 → 地图实体同步）

- **定位**：NPC 此前只在状态面板的"人物关系"里以文字呈现，不在地图上出现，世界"两张皮"。本次把 `npc_states[name].location`（文本地点）**同步成地图 `npc` 实体**，让 NPC 在地图上显示并随叙事移动。
- **核心函数**：`locateTextOnMap(map, text)`（玩家与 NPC 共用的文本→坐标定位，poi 名称优先、瓦片图例名兜底）+ `syncNpcEntitiesOnMap(gameState)`（遍历 npc_states，命中地图则 upsert `type:'npc'` 实体，未命中不挂、失去地点则清理陈旧实体）。
- **触发链路**：`applyStateChanges` 在维护 NPC 位置账本（B4/E2）后调用 `syncNpcEntitiesOnMap`，NPC 位置随每次状态变更实时刷新。
- **战争迷雾约束**：`renderer/tile-map.js` 仅在可见格绘制实体（玩家 3×3 / 战斗敌人周围），远处或未探索区域的 NPC 不显示，符合"战争迷雾"预期；NPC 不主动揭示迷雾。
- **渲染与交互**：`renderEntity` 原生支持 `npc` 类型（金色 `N` 图标）；点击地图 NPC 触发"与 X 交谈"提示并可发起 `talk` 动作（与战斗敌人点击攻击对称）。
- **测试**：`test/npc_map_check.js`（16 PASS，含 locateTextOnMap 的 poi/图例/子串/空文本定位、syncNpcEntitiesOnMap 的挂载/移动/清理/重渲染触发/与玩家不重叠）。

### 1.3 玩家世界书（手动策展 lore / 常量记忆，对标 NovelAI）

- **定位**：NovelAI 的核心卖点是"可策展世界书"——玩家对记忆有完全控制权。此前知识库只能由 AI 生成（创建世界）或 AI 抽取（一键导入 IP 设定），对玩家是黑盒。本次补上**手动策展面板**，让玩家直接增删改知识，形成「AI 生成 / AI 抽取 / 玩家手动」三路并行的记忆供给。
- **入口**：世界详情弹窗「📖 世界书」按钮；游戏内「世界」侧栏 tab 顶部「📖 编辑世界书」按钮。
- **两区**：① 知识库片段（`lore_kb.snippets`：分类 / 标题 / 内容 / 关键词，可新增 / 编辑 / 删除）；② 常量记忆（`pinned_facts`：一句话铁律，永不褪色，可新增 / 编辑 / 删除）。
- **纯函数（app-core.js）**：`addLoreSnippet` / `updateLoreSnippet` / `removeLoreSnippet` / `upsertPinnedFact` / `removePinnedFact`——统一经 `normalizeLoreSnippets` 规整字段、补 id、标题去重、空值保护，便于确定性测试。
- **预设保护**：打开世界书自动 `ensureEditableWorld()` 派生副本，不污染写死 canon（与一键导入 IP 设定一致）。
- **落库**：改动经 `saveWorlds()` 持久化，并触发 `computeEmbeddingsForWorld()` 重算向量（失败可忽略）；世界打 `manual_lore` / `manual_fact` 标记。
- **测试**：`test/world_book_check.js`（35 PASS，含纯函数单测 + 预设派生 + 全链路新增 / 编辑 / 删除 + 落库验证 + 空标题/空事实保护）。

### 1.4 性能优化（启动与每轮成本）★ 本次新增

- **定位**：此前 `computeEmbeddingsForWorld` 在初始化时遍历**所有** world 全量预跑向量（世界数多时启动卡顿），且世界脉搏（背景动态）每轮额外发一次独立 API 调用（`generateWorldPulse`），翻倍单轮成本。本次从两处降本。
- **G1 — embedding 惰性重算（app-ai.js）**：`snippetFingerprint(s)` 对 `category|title|content|keywords` 算 FNV 内容指纹；`computeEmbeddingsForWorld` 跳过「已带 embedding 且 `emb_fp` 等于当前指纹」的片段（`continue`），仅重算内容变化的片段——同时修掉"编辑片段后旧向量不更新"的隐性 bug。启动时**移除遍历所有 world 的预跑循环**，改为 `startGame` 进入具体世界时 `await computeEmbeddingsForWorld(currentWorld)` **按需重算当前世界**（新建世界则在创建时即时重算）。空 `world`/`lore_kb` 安全返回。
- **G2 — 世界脉搏合并进主响应（app-ai.js + app-game.js）**：`buildTurnUserMessage` 在提示末尾注入 `world_pulse` 可选字段说明（type: rumor|env|faction|weather + text），让 AI 在主响应里"顺手"生成一条无需玩家介入的背景动态，省去一次独立 API 调用。`currentWorld._pulseCounter` 累计推进的时间段数，达到 2 时强制提示"本回合请尽量返回 world_pulse"，避免长期无背景动态。`consumeWorldPulse(resp)`（顶层函数）优先消费主响应自带的 `world_pulse` 写入 `current_world_events`（上限 12 条，超出截断），返回 true 表示已消费、`processTurn` 即**不再降级调用 `generateWorldPulse()`**；缺失时仍降级保证脉搏不真空。
- **测试**：`test/perf_opt_check.js`（22 PASS：G1 指纹稳定性/惰性跳过/变化重算/空 world 安全；G2 consumeWorldPulse 单元/长度截断/buildTurnUserMessage 注入阈值/启动预跑移除源码断言）。

## 2. 知识库与混合 RAG

- `lore_kb.snippets` 按类别组织：规则 / 地点 / 人物 / 事件 / 物品 / 势力 / 冲突（冲突、事件各≥2 条，含触发条件）。
- **双路检索** + **行为记录滑动窗口**（最近 5 条强制注入），避免长程上下文丢失。
- `computeEmbeddingsForWorld` 预计算向量（与本地模型同格式），检索结果随对话注入；按内容指纹**惰性重算**（G1，详见 §1.4），启动时不再全量预跑。

## 3. 状态与数值系统

- **文字描述优先**：属性（胆识/灵慧/耐心/运气/意志）、关系、技能全部用文字描述，仅在显著变化时更新；兼容旧版数值的增量写法。
- **统一时间推进** `applyTimeChange`：消除 `period` 与 `current_date` 双路径相互覆盖；模式 `periods` / `continuous` / `hidden`，时段只能前进不可倒退。
- **目标系统**：主线/支线/个人，带 `deadline`；超时/条件破坏标记 `failed`（不删除，开辟新线）。
- **战斗** `CombatEngine`：D20 掷骰，HP/MP/XP，`spawn_enemies` 触发战斗，`in_combat` 状态贯穿。

## 4. 常驻记忆层 `pinned_facts`

- AI 在 JSON 的 `pinned_facts` 标注**一旦成立就永远成立、不可逆**的事实：誓言/承诺、契约/pact、诅咒/祝福、角色死亡（含 NPC）、阵营归属、不可逆世界改变、核心主线目标。
- 被**无条件全量注入每一轮对话**，AI 必须始终遵守、不可遗忘。
- `applyStateChanges` 中有状态信号兜底（死亡 / cursed / oath / pact / allegiance / deaths）自动钉住；玩家**无需手动**添加记忆。
- 详见 `docs/11_常驻记忆层设计.md`。

## 5. 预言机 / 混沌因子（oracle）

- **创建时可控**：开关 `oracle.enabled` + 混沌因子 `oracle.chaos_factor`（1-9 滑杆）。

## 7. 章节 / 幕结构（act）

- `computeCurrentAct` / `updateActProgress`：基于**已完成主线目标数 + 事件数**推导当前"幕"，跨幕时在"剧情"标签页呈现徽章与章节时间线、大事记。

## 8. 地图 ⇄ 战斗 集成（MapDataV1）

- 瓦片地图协议 `MapDataV1`，纯 CSS `div` 渲染（无图片依赖），含 `tile_legend`、`entities`、`poi`、`explored` 等字段。
- **正向链路**：点击地图空地 → BFS 寻路 → `movePlayerOnMap` 移动玩家 → 同步 `current_location` 为瓦片名。
- **战斗 ⇄ 地图**：`addMapEnemiesForCombat` 把战斗敌人显式放到图上（标记 `_combat`）；`syncMapCombatEntities` 每轮对齐（战后清场，保留 AI 叙事放置的敌人）。点地图上的敌人即攻击该敌人。
- 详见 `test/map_combat_check.js`（22 项校验）。

## 9. 反向回写（reverse-writeback）★ 本次新增

- **问题**：§8 正向链路已让"点地图→current_location 同步"，但玩家用自由文本叙事移动时（如"我走向北门"），AI 改了 `current_location`，地图上的玩家标记却停在原地——地图与叙事位置"两张皮"。
- **解决**：`applyStateChanges` 处理完 `map_data` 后调用 `syncMapPlayerFromNarrative(gameState, changes, prevLocation)`，把叙事位置写回地图。
- **定位优先级**：
  1. `changes.player_pos` 显式坐标 `{row, col}`（AI 在叙事移动时给出，**最可靠**）；
  2. `changes.current_location` 命中地图 `poi` 命名节点（名称/子串匹配）；
  3. `changes.current_location` 命中某瓦片图例名（唯一匹配时兜底）。
- **约束**：战斗中位置由战斗逻辑掌控，不回写；`prevLocation` 与新地点相同（未移动）时不回写。
- `placePlayerOnMap` 仅移动 + 揭示迷雾、**不覆盖** `current_location`；`movePlayerOnMap` 在其基础上同步 `current_location`（正向）。

## 10. 战争迷雾（fog of war）★ 本次新增

- 每张地图维护 `explored`（二维布尔网格，**持久化**进存档）与 `fog_of_war` 开关（创建时可选，默认开）。
- **三态渲染**（`TileMap.render`）：
  - 未探索 → 纯黑迷雾格（`tile-fog`，无符号、无实体）；
  - 已探索但当前不可见 → 暗淡（`opacity:0.38`，隐藏动态实体，只留地形记忆）；
  - 当前可见 → 正常渲染地形 + 实体。
- **揭示逻辑**：玩家抵达即 `revealAround` 周围 Chebyshev 半径 1；`computeFogVisible` 在**战斗中额外揭示存活 `_combat` 敌人周围**，保证战场可读。
- **稳健性**：AI 重发同尺寸地图时保留已探索进度（不每次清零）；旧存档无 `explored` 时按玩家起点补初始化。
- 详见 `test/reverse_writeback_fog_check.js`（含迷雾相关 35 项校验中的对应用例）。

## 10.5 势力声望 / 经济 / 合成 / 装备（世界模拟）★ 本次新增

把"世界模拟"从纯叙事推进到可量化的势力博弈与资源循环（对应竞品分析的⑤）：

- **多势力声望 `factions`**：每个势力独立累计 `reputation`（-100~100）与 `stance`（敌视/冷淡/中立/友善/崇敬）。AI 在 `state_changes.factions` 输出 `reputation_delta` + `stance` 驱动；状态面板「势力」标签页渲染各阵营声望条与立场颜色。
- **货币经济 `currency`**：按币种累加 delta（不允许为负），默认 `gold`；世界观可自定义币种（如灵石/银币）。状态面板「收集」页顶部显示持有货币。
- **物品/装备扩展**：`inventory` 的 `add` 支持 `type`/`equippable`/`slot`/`damage_bonus`/`ac_bonus`/`desc`。玩家可在「收集」页一键装备/卸下；**已装备武器提升攻击伤害**（接入 `action-menu.resolveAttack`），已装备盔甲在战斗数值区展示防御加成。
- **合成工坊 `crafting_recipes`**：AI 在叙事中"习得/发现"配方时通过 `state_changes.crafting.add_recipe` 注入（去重）。玩家在「收集」页合成工坊于材料齐全时一键合成——引擎消耗 `inputs` 并产出 `output`，形成"采集 → 习得 → 制作"闭环。
- **动作菜单**：新增「交易」「合成」两个按钮（非战斗、非强制），仅把意图交给 AI，实际经济行为由 AI + `currency`/`inventory` 完成。
- **AI 感知**：`buildCompactGameState` 每轮把 `factions`/`currency`/已知配方（仅 id/name，不泄露 inputs 细节）注入上下文，让 AI 校准阵营与交易行为。
- 向后兼容：旧存档无这些字段时 `applyStateChanges` 自动兜底创建；`defaultInitialState` 写入空结构。
- 详见 `test/faction_economy_check.js`（40 项校验）。

### 10.5.1 商店 NPC（固定货架 / 定价）★ 本次新增

把"交易"从纯 AI 叙事即时处理，补上可量化、可复玩的店铺系统（对应竞品分析的⑤延伸）：

- **数据归属**：`world.shops[]` 为世界级店铺（随世界存档），每个店铺含 `id`/`name`/`owner`/`location`/`currency` 与固定 `stock[]`（`item_id`/`name`/`price`/`count`/`type` 等）。`loadWorlds()` 在迁移时为尚无店铺的世界注入一个通用「杂货铺」（治疗药水/干粮/火把/绳索）。
- **购买 `buyFromShop(gs, world, shopId, itemId, qty)`**：校验店铺存在、货架有货、玩家该币种货币充足 → 扣货币 + 减库存 + 入背包（已存在则叠加 count）。价格以货架 `price` 为准（固定定价）。

## 10.10 基调显式持久化（H）★ 本次新增

旧版 `buildToneGuide()` 仅靠关键词在 name/desc/hero 里猜「日常/高张力/悬疑/浪漫」，且同一世界全程不变（竞品矩阵 ⑩「AI 猜、全程不变」）。本次改为：**世界生成时由 AI 显式定调并持久化到 `currentWorld.tone`，剧情中也可由事件切换基调**，从而让 system prompt 的节奏控制更精准。

- **纯函数（app-core.js，便于确定性测试）**：
  - `normalizeTone(raw)`：把 AI 返回对象 / 字符串 / 旧式数组统一规范为 `{ primary, labels, description }`；非法/空输入返回 `null`（触发兜底）。
  - `inferToneFromWorld(world)`：旧版关键词正则推断（desc/hero/opening_narrative），作为 AI 未显式定调时的兜底，**旧世界向后兼容**。
  - `buildToneGuideString(toneObj)`：把基调对象渲染为注入 system prompt 的多行指引（与旧 `buildToneGuide` 格式保持一致，含四向叙事指引）。
  - `resolveToneGuide(world)`：优先用持久化 `world.tone` 生成指引，否则降级为 `inferToneFromWorld` 推断。
- **接入**：
  - 世界生成 `buildWorldGenerationPrompt` 新增**必填**字段 `tone`（primary/labels/description），由 AI 显式判定；`callWorldGenerationLLM` 原样返回；`generateWorld` 将 `normalizeTone(generated.tone) || inferToneFromWorld(...)` 写入 `world.tone` 并随 `saveWorlds()` 持久化。模拟模式 `mockGenerateWorld` 按题材返回合理 tone。
  - `buildSystemPrompt` 改用 `resolveToneGuide(currentWorld)` 取基调——**运行时恒定采用持久化 tone**，不再每次靠关键词猜。
  - **剧情中切基调**：`applyStateChanges` 新增 `changes.tone` 分支——AI 可在 `state_changes` 里显式改变当前基调，更新 `currentWorld.tone`、调用 `saveWorlds()` 持久化、并 `invalidateSystemPromptCache()` 让下一轮 system prompt 采用新基调；若主基调变化还经 `addBehaviorRecords` 留下「故事基调悄然转变：X → Y」的可见记录。
  - **UI**：世界卡片与世界详情弹窗新增「叙事基调」展示（卡片为蓝色 `tone` 标签），旧世界无 tone 时仅靠推断、不报错。
- **测试**：`test/tone_persistence_check.js`（14 PASS，含 normalizeTone 规范化 / resolveToneGuide 显式优先+推断兜底 / mockGenerateWorld 各分支 tone / applyStateChanges 切基调更新+持久化+刷新缓存）。

## 10.11 API URL 拼接健壮性（I）
- **问题**：原 `buildApiUrl` 仅在 baseUrl 末尾去除尾斜杠后硬拼 `/chat/completions`，存在多重脆弱点：baseUrl 已含 `/chat/completions` 或 `/v1/chat/completions` 会**双重拼接**（`/chat/completions/chat/completions`）；缺协议（如 `api.deepseek.com`）被当作相对路径；CORS 代理拼接未做规范化。
- **实现**：将 `buildApiUrl` 与 `normalizeApiBaseUrl` 抽为 `js/app-core.js` 的纯函数（app-core 先于 app-ui/app-ai 加载，全局唯一），并增强健壮性：
  - `normalizeApiBaseUrl`：`trim` → 协议补全（localhost / 127.0.0.1 / IP 用 `http://`，含 `.` 的域名补 `https://`）→ **循环剥离**已存在的 `/chat/completions`（兼容尾斜杠 / 大小写 / 重复拼接）→ 去尾斜杠。
  - `buildApiUrl`：先规范化 base 再拼 `/chat/completions`；若配置 CORS 代理，则对代理同样做 `trim` / 去尾斜杠 / 协议补全后前缀转发（`proxy + "/" + apiPath`）。
  - 空 / 空白 baseUrl 返回空串（调用处已对空串抛清晰错误），不再产生 `/chat/completions` 这类无效 URL。
- **测试**：`test/url_build_check.js`（25 PASS，覆盖裸域名 / 尾斜杠 / 已含端点去重 / 双重拼接 / OpenAI `/v1` / 协议补全 / localhost / 127.0.0.1 / CORS 代理转发与代理尾斜杠与代理协议补全 / 代理空白 / 空与空白 / 前后空格，以及 `normalizeApiBaseUrl` 单独边界）。
