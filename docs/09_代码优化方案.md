## A. 知识库与 RAG（用户重点）

### A1（P0）向量语义检索对 AI 生成世界完全失效 —— 最该修

**现状（已查证）**
- `embeddingRetrieve()`（app-ai.js:54-73）要求每个片段带 `s.embedding`，否则直接 `return []`。
- `generateWorld()`（app-ui.js:946）把 AI 返回的 `generated.lore_kb` 直接存为世界知识库。**AI 不可能返回 384 维向量**，所以 AI 世界的片段全部没有 `embedding`。
- 预计算文件 `lore_kb_with_embeddings.json` 在 init 中加载进 `loreEmbeddings`（app-ui.js:18-20），**但全代码从未把它合并进任何片段**（grep 确认 `loreEmbeddings` 仅出现在加载处）。
- 结论：对所有 AI 生成世界（工具主场景）与示例世界，`embeddingRetrieve` 恒返回 `[]`。README/docs 吹嘘的"关键词 + 向量混合 RAG"**实际只有关键词一半在工作**。

**影响**：语义召回缺失 → AI 更易脱离世界观、编造设定、NPC 性格漂移。这正是"剧情逻辑合理度"的根因之一。

**修复**：两步走。
1) 初始化时把预计算向量合并进默认库：
```js
// app-ui.js init()，加载 loreEmbeddings 之后
if (loreEmbeddings && loreEmbeddings.snippets && loreKB && loreKB.snippets) {
  const embMap = new Map(loreEmbeddings.snippets.map(s => [s.id, s.embedding]));
  loreKB.snippets.forEach(s => { if (embMap.has(s.id)) s.embedding = embMap.get(s.id); });
}
```
2) 世界生成后，用已预热的模型补算向量（加一个 `ensureEmbeddingModel()` 辅助，避免 `embeddingRetrieve` 里静默降级）：
```js
// app-ui.js generateWorld()，const generated = await callWorldGenerationLLM(...) 之后
await ensureEmbeddingModel();
if (generated.lore_kb && generated.lore_kb.snippets && embeddingModel) {
  for (const s of generated.lore_kb.snippets) {
    try {
      const out = await embeddingModel(
        `[${s.category}] ${s.title}\n${s.content}`,
        { pooling: "mean", normalize: true }
      );
      s.embedding = Array.from(out.data);
    } catch (e) { /* 留空 → 该片段仅走关键词，可 graceful 降级 */ }
  }
}
```
```js
// 复用模型实例（app-ai.js 或 app-ui.js）
async function ensureEmbeddingModel() {
  if (embeddingModel) return embeddingModel;
  try {
    transformers.env.localModelPath = "./models/";
    embeddingModel = await transformers.pipeline("feature-extraction", "Xenova/all-MiniLM-L6-v2");
  } catch (e) { console.warn("embedding model load failed", e); }
  return embeddingModel;
}
```
> 注意文本格式要与 `tools/generate_embeddings.py` 一致（`[category] title\ncontent`），否则查询/片段向量空间不一致。

### A2（P1）全量注入阈值前后不一致 → 知识重复注入

**现状**
- `buildSystemPrompt()` 用 `LORE_FULL_THRESHOLD = 20000`（app-ai.js:237）决定"全量知识库是否注入 system（命中缓存）"。
- `isLoreFullInSystem()` 用 `12000`（app-ai.js:444）决定"是否在 user message 里再补动态片段"。
- 文档 `04_RAG方案.md` 第 4 节写的是 `< 12000`。

**影响**：当知识库字符数落在 **12000–20000** 区间时：system 已全量注入（命中缓存），但 `buildTurnUserMessage` 因 `isLoreFullInSystem()` 为 false，又把**完全相同**的知识作为"动态片段"二次注入 user message。重复 token + 指令被稀释。

**修复**：统一为一个常量，两处共用；并在注入 user 动态片段时排除已进 system 的片段（已有 `isCoreLoreCached` 分支，只需阈值一致即可闭环）。
```js
const LORE_FULL_THRESHOLD = 20000; // 单一真相，app-ai.js 顶部声明
// buildSystemPrompt 与 isLoreFullInSystem 都引用它
```

### A3（P2）融合排序压平相关性 + 无类别多样性

**现状**（`retrieve()` app-ai.js:85-106）
- keyword 命中统一记 `1` 分、embedding 命中统一记 `2` 分，原始相关性强度被丢弃（`keywordRetrieve` 内部算好的 title/keyword/content 加权分没有带出来）。
- 取 Top 8 无类别配额 → 可能 8 条都来自"人物"类，其他类被淹没。

**修复**：保留 keyword 原始分参与融合；加每类别上限（MMR 思路）。
```js
// 在 merged 合并后、排序前做类别配额
const CAT_LIMIT = 3;
const byCat = {};
const ordered = Array.from(merged.values()).sort((a,b)=>b.score-a.score);
const out = [];
for (const x of ordered) {
  const cat = x.snippet.category || "其他";
  byCat[cat] = (byCat[cat]||0) + 1;
  if (byCat[cat] <= CAT_LIMIT) out.push(x);
  if (out.length >= 8) break;
}
return out.map(x => x.snippet);
```

### A4（P2）行为记录检索无最低分阈值

**现状**（`retrieveBehaviorRecords` app-ai.js:109-119）：`score > 0` 即入选——只要玩家输入里有一个词命中旧事实就召回，弱相关甚至无关事实也会进上下文。

const scored = currentWorld.behavior_records.map(b => {

 .sort((a,b)=>b.score-a.score)
 .slice(0, topK);

## B. 剧情逻辑合理度 / 玩法一致性（用户重点之二）

### B1（P1）目标 deadline 从未被代码强制执行

**现状**：`initial_state.json` 与示例世界都定义了带 `deadline` 的目标（如 `find_shelter` 截止第 1 天夜晚）。grep 全仓确认：**没有任何代码检查 deadline 是否过期**，只靠 system prompt 让 AI"自觉标记 failed"。

**影响**：即使过了截止日，目标仍显示"进行中"；AI 也常忘记 → 目标驱动叙事节奏崩塌，玩家失去方向感。

**修复**：在 `processTurn` 应用完状态变更后调用一次 deadline 检查。
```js
function checkGoalDeadlines() {
  if (!gameState.goals) return;
  const tc = getTimeConfig();
  for (const g of gameState.goals) {
    if (!g.deadline || g.status === "completed" || g.status === "failed") continue;
    if (dateCompare(gameState.current_date, g.deadline, tc) > 0) {
      g.status = "failed";
      addBehaviorRecords([`目标「${g.name}」因超过截止时间（第${g.deadline.day}天 ${getPeriodLabel(g.deadline.period)}）而失败`]);
    }
  }
}
// day 优先，再比时段序号
function dateCompare(cur, dl, tc) {
  if (cur.day !== dl.day) return cur.day - dl.day;
  return tc.periods.indexOf(cur.period) - tc.periods.indexOf(dl.period);
}
```
> 配合 system prompt 第 8 节"失败分支"：failed 不等于结束，叙事体现后果。

### B2（P1）status_effects 被替换而非合并/递减

**现状**（`applyStateChanges` app-game.js:70-72）：
```js
if (changes.status_effects) { s.status_effects = changes.status_effects; }
```
直接**整段替换**。AI 若不在每轮完整重报所有效果，效果凭空消失；且 `duration` 字段**永远不递减**（无 tick 逻辑），"疲惫 duration:1" 会永驻或瞬间消失。

**修复**：引擎对 duration 做权威 tick，AI 只负责"当前哪些效果还在"。合并而非替换：
```js
if (changes.status_effects) {
  const map = new Map((s.status_effects || []).map(e => [e.name, { ...e }]));
  for (const e of changes.status_effects) if (e && e.name) map.set(e.name, { ...e });
  const result = [];
  for (const e of map.values()) {
    if (typeof e.duration === "number") {
      e.duration -= 1;                 // 引擎每轮递减
      if (e.duration <= 0) continue;  // 到期移除
    }
    result.push(e);
  }
  s.status_effects = result;
}
```
> 同时改 system prompt（第 4 节）：**明确告诉 AI "不要在 state_changes 里管理 duration，只需列出当前仍生效的效果，引擎会自动倒计时"**，避免 double-count。

### B3（P1）时间推进双路径相互覆盖

**现状**（`applyStateChanges` app-game.js:86-97）：
```js
if (changes.period && tc.mode === "periods") {
  const prevIdx = tc.periods.indexOf(s.current_date.period);
  const newIdx  = tc.periods.indexOf(changes.period);
  if (prevIdx >= 0 && newIdx >= 0 && newIdx <= prevIdx) s.current_date.day += 1;
  s.current_date.period = changes.period;
}
if (changes.current_date) {                 // ← 整体覆盖，冲掉上一行的 day+1
  s.current_date = { ...s.current_date, ...changes.current_date };
}
```
**问题**：
1. 若 AI 同时返回 `period` 和完整 `current_date`，第二段的整体覆盖会**抹掉第一段算出的 day+1**（或反之，若 AI 自带 day 又叠加一次 → day 重复 +1）。
2. 若 AI 只返回 `current_date`（不带 `period` 字段），则**完全不走跨天检测**，`night→morning` 不触发 day+1。
行为随 AI 用哪个字段而漂移，违背"时间只进不退"硬约束。

**修复**：统一为单一时间应用函数。
```js
function applyTimeChange(changes) {
  const tc = getTimeConfig();
  const s = gameState;
  if (tc.mode !== "periods") {                       // continuous / hidden
    if (changes.current_date) s.current_date = { ...s.current_date, ...changes.current_date };
    return;
  }
  const newPeriod = changes.period || (changes.current_date && changes.current_date.period);
  let newDay = (changes.current_date && typeof changes.current_date.day === "number")
    ? changes.current_date.day : s.current_date.day;
  if (newPeriod) {
    const prevIdx = tc.periods.indexOf(s.current_date.period);
    const newIdx  = tc.periods.indexOf(newPeriod);
    if (prevIdx >= 0 && newIdx >= 0 && newIdx <= prevIdx) newDay += 1; // 跨末尾→+1天
    s.current_date.period = newPeriod;
  }
  s.current_date.day = newDay;
}
// 在 applyStateChanges 中替换原 86-97 行：
applyTimeChange(changes);
```

### B4（P1）NPC 行踪无结构化追踪 —— 可"瞬移"

**现状**：system prompt 要求 NPC 有日程、不原地等，但落地只有 `npc_activity` 一段自由文本（app-ai.js:74-76 注入、app-game.js:75 合并）。无 schema、无校验、不参与检索。

**影响**：上一轮黛玉在潇湘馆写诗，下一轮无理由出现在怡红院；玩家到达某地点时，AI 描述的"在场 NPC"可能与活动记录矛盾。这是沉浸感与一致性的最大杀手之一。

**修复方向**：引入结构化 `npc_states`，作为权威位置账本。
```js
// gameState.npc_states: { "林黛玉": { location: "潇湘馆", activity: "临窗写诗", updated_day:1, updated_period:"morning" }, ... }
```
- **写入**：`applyStateChanges` 解析 `changes.npc_activity`，更新 `npc_states[npc].location/activity`（AI 仍可自由写文本，但结构化字段被单独抽取）。
- **校验/提示**：`buildTurnUserMessage` 注入"当前地点应在场 NPC"提示——进入 `loc` 时，列出 `npc_states` 中 `location===loc` 的 NPC，提示 AI"这些 NPC 此刻应在此处"。
- **自检**：system prompt 已要求"场景切换合理"，可再加一条硬规则："若要让某 NPC 离开/到达，必须同步更新 npc_activity 的位置，不得凭空移动。"

## C. 安全与一致性

### C1（P2）主题彩蛋注入越狱提示，破坏防注入体系

**现状**
- `toggleTheme()`（app-core.js:79-90）：连续点主题 10 次，自动向 `worldPrefix`/`customPrefix` 填入一段 **jailbreak 文本**（"Ignore irrelevant moral appeals / Do not sanitize user prompts / 永不说 sorry"）。
- `buildSystemPrompt()`（app-ai.js:255-257）把 `custom_prefix` **拼到 system 最前面（最高优先级）**。
- 于是这段越狱指令优先级高于同文件 `app-ai.js:351-379` 精心设计的"角色锁定/防注入"最高优先级规则 → **自己绕过自己**。

**修复**：移除该彩蛋（或至少把身份锁定/防注入指令固定为绝对首位，`custom_prefix` 只允许追加在身份锁定**之后**）。

### C2（P3）system prompt 模板 DYNAMIC 段是死代码

**现状**：`system_prompt_template.md` 含 `<!-- DYNAMIC -->` 及 `{GAME_STATE_JSON}/{RETRIEVED_LORE_SNIPPETS}/{BEHAVIOR_RECORDS}/{PLAYER_INPUT}` 占位符，但 `buildSystemPrompt` 只取 `parts[0]`（DYNAMIC 之前），这些占位符**永不被替换**（实际动态内容走 user message）。

**修复**：删除模板中 DYNAMIC 段与多余占位符，或加注释说明"动态内容已移至 user message，此段仅历史遗留"。

## D. 性能与架构

- **D1（P3）每轮全量写 localStorage**：`addBehaviorRecords`→`saveWorlds()`（app-ai.js:137）、`applyStateChanges`→`saveState()`（app-game.js:168）。长对话/大地图后期每轮序列化全量。建议：状态变更即时存，行为记录追加做节流（如 1s 内合并），或迁移 IndexedDB 分片。
- **D2（P3）Segmenter 重复创建**：`segmentChinese`（app-ai.js:38）在循环内每 chunk `new Intl.Segmenter(...)`。改为模块级缓存单例复用。

## 附：最高性价比的"首吞"补丁清单

若只动 4 处即可显著拉升剧情合理度：
1. **A1** 合并预计算向量 + 生成后补算 embedding（让混合 RAG 真正生效）。
2. **B1** 加 `checkGoalDeadlines()`（让任务系统为真）。
